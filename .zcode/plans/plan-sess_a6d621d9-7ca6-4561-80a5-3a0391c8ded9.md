## 1. NEW `WhiteBalanceCalculator.kt`
A pure, testable helper:
```kotlin
object WhiteBalanceCalculator {
    data class RgbGains(val r: Float, val g: Float, val b: Float)

    /** App temp (-2..+2) → Kelvin (9000K cool .. 2300K warm, 5600K at 0). */
    fun tempToKelvin(temp: Float): Float

    /** Kelvin → RGB white-balance gains via Planckian-locus → CIE 1960 uv
     *  → XYZ → linear sRGB, normalized so green = 1.0 and max gain ≤ ~2.5. */
    fun kelvinToGains(kelvin: Float): RgbGains

    /** App temp → gains (convenience). */
    fun tempToGains(temp: Float): RgbGains
}
```
Algorithm (standard, well-documented):
- Kelvin → (u, v) on the Planckian locus using the Robertson/McCamy approximation.
- (u, v) → XYZ (matrix from CIE 1960 UV).
- XYZ → linear RGB (sRGB matrix).
- Normalize: divide all channels by G so green = 1.0; clamp each to a safe ceiling (e.g. 2.5) so extreme values don't blow out highlights.
- At 5600K the gains are ~neutral (R≈1.0, B≈1.0); warm raises R / lowers B; cool raises B / lowers R.

## 2. `PreviewSessionManager.bindPreview` — gains on the CaptureRequest
Add a `temperature: Float` parameter. Inside, compute gains and apply to **both** the Preview and ImageCapture builders via the existing `Camera2Interop.Extender`:
```kotlin
val gains = WhiteBalanceCalculator.tempToGains(temperature)
val rggb = RggbChannelVector(gains.r, gains.g, gains.g, gains.b)

Camera2Interop.Extender(previewBuilder).apply {
    setCaptureRequestOption(CONTROL_AWB_MODE, CONTROL_AWB_MODE_OFF)
    setCaptureRequestOption(COLOR_CORRECTION_MODE, COLOR_CORRECTION_MODE_TRANSFORM_MATRIX)
    setCaptureRequestOption(COLOR_CORRECTION_GAINS, rggb)
}
```
Same for the ImageCapture builder. Because these are baked into the session's CaptureRequest, the **JPEG capture inherits the WB** — no post-processing needed for temperature.

Notes:
- `CONTROL_AWB_MODE_OFF` is required or the gains are ignored.
- These calls are no-ops on devices without `MANUAL_POST_PROCSENSING` — the request simply drops the keys, no crash (your chosen "no fallback" behavior: slider does nothing visible there).
- Wrap each `setCaptureRequestOption` in try/catch so one unsupported key doesn't abort the whole bind.

## 3. `CameraPreviewView` — live WB updates (no rebind)
Add a `LaunchedEffect(temperature, camera)` mirroring the existing exposure one. On temperature change, push the new gains through **`Camera2CameraControl`** so the live preview updates smoothly without restarting the camera:
```kotlin
LaunchedEffect(temperature, camera) {
    camera?.let { c ->
        val c2c = Camera2CameraControl.from(c.cameraControl)
        val gains = WhiteBalanceCalculator.tempToGains(temperature)
        c2c.captureRequestOptions = CaptureRequestOptions.Builder()
            .setCaptureRequestOption(CONTROL_AWB_MODE, CONTROL_AWB_MODE_OFF)
            .setCaptureRequestOption(COLOR_CORRECTION_GAINS, RggbChannelVector(...))
            .build()
    }
}
```
`Camera2CameraControl` applies the options to the active repeating (preview) request in place — same pattern the codebase already uses for exposure index. Forward `temperature` into the `bindPreview(...)` call too so still captures at trigger time have the right value.

## Trade-offs you're accepting (per your "Make WB real, no fallback" choice)
- **Unsupported devices:** on phones lacking `MANUAL_POST_PROCSENSING` (common on lower-end / older devices, and some logical multi-camera setups), the slider will have no visible effect. No crash, just no-op.
- **Auto WB is disabled** while the panel is in use — at `temp = 0` I'll set gains to neutral (D56 daylight) rather than re-enabling auto, so behavior is predictable and matches the readout.
- **Front camera** uses the default CameraX bind path (not `PreviewSessionManager`); I'll also wire WB there via the same `Camera2CameraControl` live-update so it works on the front camera too, but the still-capture bake-in won't apply there (front camera captures would only get preview-side gains via the repeating request, which CameraX still captures inherit in most cases).
